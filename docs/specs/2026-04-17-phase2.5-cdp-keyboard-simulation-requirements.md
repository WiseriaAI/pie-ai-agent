# Phase 2.5 — CDP Keyboard Simulation for Canvas-Rendered Editors

## Problem Frame

Phase 2 Agent 用 DOM 操作（`execCommand`、native value setter、InputEvent）实现 type 工具。对大多数富文本编辑器（Slate/Quill/ProseMirror/Lexical）和普通输入框都有效，但**对 canvas-rendered editor 完全失效**：

- **飞书 Docs**（新版 Docx 引擎）
- **Google Docs**（Kix 引擎）
- **Notion**（部分编辑 surface）
- 任何把文字画到 `<canvas>` 而不是 DOM 的编辑器

这些编辑器共享的架构：
1. 可见文档是 `<canvas>` 或独立渲染树
2. 用一个**隐藏 `<textarea>`/`<input>`** 作为 IME + 键盘捕获器
3. 编辑器只消费 `isTrusted: true` 的键盘/composition 事件
4. Synthetic event（JS 合成的）被当 noise 丢弃

**Phase 2 现状（commit `219069b`）**：Agent 会识别这种情况并**优雅失败**（通过"IME buffer heuristic"），不再产生假 success。但**用户想在飞书 / Google Docs 里让 Agent 打字的场景完全没法做**。

**Phase 2.5 目标**：引入 Chrome DevTools Protocol（CDP）的 `Input.insertText` 和 `Input.dispatchKeyEvent`，让 Agent 能模拟**真实键盘输入**（`isTrusted: true`），穿透这类编辑器的防线。

## Requirements

### 核心能力

- R1. 通过 `chrome.debugger` + CDP `Input.insertText` 向当前 tab 发送 trusted keyboard input，canvas 编辑器识别为真实用户输入并正确渲染
- R2. 支持特殊键（Enter / Tab / Escape / Backspace / Arrow keys）通过 `Input.dispatchKeyEvent` 发送
- R3. 调用前确保目标元素 focus（依赖已有 `click` 工具先定位到编辑器区域）
- R4. 每个任务级别 attach 一次 debugger，任务结束（done/fail/abort/disconnect）立即 detach

### 用户授权模型

- R5. manifest 新增 `debugger` 权限，浏览器首次运行时弹出授权确认
- R6. Settings 中添加"启用键盘模拟"开关（默认 **关闭**），用户明确知晓后开启
- R7. 开启时 Settings 显示警告："使用键盘模拟时 Chrome 会显示黄色调试条。每次任务期间扩展可通过 CDP 完全控制当前 tab"
- R8. 开关关闭时，keyboard 工具对 Agent 隐藏（不注册到 tools 列表），Agent 尝试调用 type 失败时提示用户"如需在飞书等 canvas 编辑器中输入，请在 Settings 启用键盘模拟"

### 工具模型

- R9. 新增独立工具 `dispatch_keyboard_input(text, after_element_index?)`
  - `text`: 要输入的字符串
  - `after_element_index`: 可选，先 click 定位到该元素再输入（作为 focus 保证）
- R10. 新增独立工具 `press_key(key)` 用于 Enter/Tab/Escape 等单键
- R11. **不**替换现有 `type` 工具。`type` 继续走 DOM 路径（大多数场景更快更轻量）；Agent 根据编辑器类型选择或在 type 失败后升级到 keyboard

### 安全与边界

- R12. Risk 分级：`dispatch_keyboard_input` 默认 **high risk**（总要确认），因为 CDP 绕过所有 DOM 可见性和可交互性校验
- R13. Debugger attach 时发 `agent-step` 事件告知 Panel "已进入键盘模拟模式"，任务结束时发 "已退出"
- R14. `chrome.debugger.onDetach` 监听用户点黄条"取消"：立即中止 Agent 任务并发 `agent-done-task { success: false, summary: "用户取消调试授权" }`
- R15. Detach 必须 bulletproof：Promise finally、port onDisconnect、AbortSignal 任何一条终止路径都必须执行 detach

### UX 边界

- R16. 一个 tab 一次只能有一个 debugger。如果用户已在目标 tab 开着 DevTools，attach 会失败 → 向用户返回明确错误提示"请先关闭 DevTools"
- R17. 黄色调试条是浏览器级 UI，**不可隐藏或绕过**——这本身就是安全信号
- R18. Agent 任务结束后 debugger 必须 detach，不能长驻（用户不应看到黄条持续存在）

## Success Criteria

- 在飞书 Docs / Google Docs / Notion 真实文档里让 Agent 输入文字，可见内容正确出现
- 开关默认关闭时，用户完全不感知 debugger 权限存在；开启时行为符合预期且每次任务后正确清理
- 黄色调试条只在 Agent 任务**实际执行键盘模拟**期间出现，不是 Settings 开启后常驻
- 用户点黄条取消授权时，Agent 立即知道并终止

## Scope Boundaries

- **不包含**鼠标事件模拟（`Input.dispatchMouseEvent`）——作为 Phase 2.6 的独立决策
- **不包含**复制/粘贴（`Input.dispatchKeyEvent` + `Ctrl+V` 组合）——延后
- **不包含**跨 tab 键盘输入——debugger 绑定 tabId，改 tab 前必须 detach/re-attach
- **不包含** CDP `Input.insertText` 以外的 CDP 能力（DOM API、Network API 等）
- **不包含** chrome.debugger 的 keep-alive 心跳（5 min idle 限制依然适用）
- **不包含** crash recovery——SW 重启时 debugger 自动 detach（Chrome 行为），用户任务被动中止

## Key Decisions

- **独立工具而非 type 的 fallback**：让 Agent 显式决定是否升级到 CDP 路径。`type` 保持轻量快速（不 attach debugger），只有 Agent 在 type 失败后才调 `dispatch_keyboard_input`。理由：Agent 可根据编辑器指纹（Phase 2 已识别 Feishu/Google/Notion）前置选择；避免隐式的 debugger attach 让用户惊讶。

- **每任务 attach，任务结束 detach**（而非全局 attach 或每工具 attach）：平衡 UX（黄条只在执行期间）和性能（attach 有 ~100-300ms 成本）。任务期间多次 keyboard tool 调用共享同一个 debugger 会话。

- **默认关闭 + Settings 显式开启**：CDP 权限比 `<all_urls>` DOM 访问严重得多——debugger 能读 Network、全局 JS state、Cookie。用户必须明确理解交换的是什么。关闭时工具对 Agent 不可见，避免 LLM 认为有这个选项而误导用户。

- **dispatch_keyboard_input 总是 high risk**：CDP 发送的事件绕过所有 DOM 可见性/残障/readonly 校验，理论上能触发任何键盘绑定逻辑。保守策略：每次调用都走 Agent confirm 卡，避免 Agent 在不明情况下批量输入。

- **onDetach 必须被处理**：用户点黄条"取消"是 Chrome 唯一的用户退出机制。不监听 onDetach → Agent 会继续尝试 sendCommand → 抛错 → 可能进入错误恢复循环。

## Dependencies / Assumptions

- Phase 2 已完成（Agent Loop + Port 协议 + Risk 分级就位）
- `chrome.debugger` API 在 Chrome Extension MV3 下可用（确认：Chrome 118+ 支持，且是 sideeffect-free 可声明权限）
- 目标 tab 上没有其他 debugger 客户端（用户 DevTools 开着时降级失败）
- 用户愿意接受黄色调试条——这是产品决策，可能影响采纳率

## Outstanding Questions

### Resolve Before Planning

- [Affects R6-R8][Product] Settings 的"键盘模拟"开关是否只是二元（开/关），还是要支持"每次询问"模式（任务开始时弹确认）？后者 UX 更友好但增加一个 UI state
- [Affects R13-R14][UX] Agent 执行键盘模拟期间 Side Panel 是否要有持续的视觉标识（比如红色 border / banner "键盘模拟中"）？还是只靠浏览器黄条

### Deferred to Planning

- [Affects R1][Technical] `Input.insertText` vs `Input.dispatchKeyEvent` 的选择——前者一次插入一段文字，后者是单个 keyDown/keyUp。对 composition（中文输入）需要验证哪个在飞书/Google Docs 上行为正确
- [Affects R2][Technical] 特殊键的 keyCode / windowsVirtualKeyCode 映射表（Enter=13, Tab=9, Escape=27 等）
- [Affects R3][Needs research] `Input.insertText` 是否自动 focus，还是需要先 `Runtime.evaluate` 调 element.focus()
- [Affects R16][Technical] 检测 DevTools 已打开的方式——尝试 attach 如果失败则降级？还是有预检 API
- [Affects R4][Technical] SW 意外重启时 debugger 是否自动 detach 由 Chrome 保证？需要实验确认，如果不保证需要 onInstalled 时清理残留 attach
